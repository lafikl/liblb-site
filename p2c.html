What is The Power Of Two Random Choices?

In centerlalized load balancers world, implementing a least loaded policy is very simple. Because it has a complete view of the cluster, since all the traffic goes through it.
It can do it by keeping counters of open connections for every host, and route traffic to the least loaded one.
But once your systems outgrows that load balancer, this could never work as is.

First, we need a way to periodically propagate the load information of the hosts to all the load balancers.
Once we do so, the load of the servers will look like a roller coaster, moving up and down.
It's because that the load information distrubted will always be stale.
Which means that the load balancers, are going to route more traffic than it should to the least loaded server.
This will happen again, when it gets the new load information with the newest least loaded host.
While the previously least loaded host becomes idle, developing a herd-like behavior.

To address this issue, the authors of The Power of Two Random Choices: A Survey of Techniques and Results"
observed that picking hosts randomly would result in better load distribution O(log n) where n is the number of servers.

Deriving from that fact, they devised a strikingly simple solution that would solve the issue of the traffic fluctuation.
It works by choosing two hosts either via hashing or randomly and then pick the least loaded of the two.
The algorithm gaurantees that the max load of a server is O(log log n), where n is the number of servers.

Client Example

An example of the code used by the client to assign requests to servers, using liblb/p2c Go package, which implements The Power Of Two Random Choices algorithm.

// Power of Two choices example
lb := p2c.New("127.0.0.1:8009", "127.0.0.1:8008", "127.0.0.1:8007")
for i := 0; i < 10; i++ {
    // uses random power of two choices, because the key length == 0
    host, err := lb.Balance("")
    if err != nil {
        log.Fatal(err)
    }
    // load should be ~33% per host
    fmt.Printf("Send request #%d to host %s\n", i, host)
    // when the work assigned to the host is done
    lb.Done(host)
}

Strengths

  • Aware of actual load of the servers, and distributes traffic accordingly.
  • Very easy to implement in a couple of lines.
  • Guarantees a server's load to be O(log log n) where n is the number of servers.

Weaknesses

  • The application needs to implement a way to propagate the load informations to the load balancers.
  • It has no way of assigning certain requests to certain servers (affinity), which makes it not suitable for systems that utilizes caches to avoid doing heavy computations over and over.

Use Cases

  • Used by Facebook's Scuba system, to distrbute writes to the servers with the most free disk space.
  • Distributed Load Balancers. Such as ELB, CloudFlare Load Balancer
  • Client's load balancer in microservices architecture. Such as Finagle
